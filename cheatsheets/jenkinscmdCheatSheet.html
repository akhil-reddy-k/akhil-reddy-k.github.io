Pipeline Overview

A Jenkins Pipeline is automation logic written as code in a Jenkinsfile, committed to your repository. It defines how your code is built, tested, and deployed.

Key Insight
Pipeline-as-Code means your CI/CD lives in Git — versioned, reviewed, and auditable alongside your app code.
Two Syntax Styles:
Declarative Structured, opinionated, easier to learn. Uses pipeline{} block. Recommended
Scripted Groovy-based, flexible, full programming power. Uses node{} block.
Jenkinsfile · Declarative Skeleton 
pipeline {
  agent any

  environment {
    APP_NAME = 'my-app'
  }

  parameters {
    string(name: 'BRANCH', defaultValue: 'main')
  }

  triggers {
    cron('H 2 * * *')
  }

  stages {
    stage('Build') {
      steps {
        sh 'npm install'
      }
    }
    stage('Test') {
      steps {
        sh 'npm test'
      }
    }
  }

  post {
    always { echo 'Done!' }
    failure { mail to: 'qa@co.com' }
  }
}

03

Agent Directive

The agent directive tells Jenkins WHERE to run your pipeline — on which machine, container, or pod.

🌐 agent any
Run on any available agent. Good for simple setups. 
agent any
🚫 agent none
No global agent. Define agent per stage. Use when different stages need different environments. 
agent none
// then per stage:
stage('Test') {
  agent { label 'linux' }
}
🏷 agent label
Run on a specific agent by label (e.g. linux, windows, mac, gpu-node). 
agent {
  label 'linux-agent'
}
🐳 agent docker
Run inside a Docker container. Container is spun up, pipeline runs, then destroyed. 
agent {
  docker {
    image 'node:18-alpine'
    args '-v /tmp:/tmp'
  }
}
☸️ agent kubernetes
Run in a Kubernetes pod. Scale Jenkins agents dynamically with K8s. 
agent {
  kubernetes {
    yaml '''
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: node
    image: node:18
'''
  }
}
📦 agent dockerfile
Build from a Dockerfile in your repo and use it as the agent. 
agent {
  dockerfile {
    filename 'Dockerfile.ci'
    dir 'docker'
  }
}

Stage with All Options
Full Stage Syntax 
stage('Run Tests') {
  // Which agent for this stage
  agent { label 'test-node' }

  // Only run if condition is true
  when {
    branch 'main'
  }

  // Stage-level environment
  environment {
    TEST_ENV = 'staging'
  }

  // Stage options
  options {
    timeout(time: 30, unit: 'MINUTES')
    retry(2)
  }

  // What to do
  steps {
    sh 'npx playwright test'
  }

  // After this stage
  post {
    always {
      publishHTML([
        reportDir: 'playwright-report',
        reportFiles: 'index.html',
        reportName: 'Test Report'
      ])
    }
    failure {
      echo 'Tests failed!'
    }
  }
}
💡 Pro Tip
Use stash / unstash to share files between stages running on different agents.

Defining & Using Variables
environment{} block 
pipeline {
  agent any

  environment {
    // Static values
    APP_NAME  = 'my-app'
    REGISTRY  = 'docker.io/myorg'

    // From credentials store
    API_KEY   = credentials('api-key-id')

    // Dynamic - using sh output
    GIT_SHA   = sh(script: 'git rev-parse --short HEAD',
                      returnStdout: true).trim()
  }

  stages {
    stage('Build') {
      steps {
        // Access with ${} in strings
        sh "docker build -t ${REGISTRY}/${APP_NAME}:${GIT_SHA} ."

        // Or env.VARIABLE in Groovy
        echo "Build: ${env.BUILD_NUMBER}"
      }
    }
  }
}

Parameters — All Types

All Parameter Types 
parameters {

  // Free text input
  string(
    name: 'VERSION',
    defaultValue: '1.0.0',
    description: 'App version to deploy'
  )

  // Multi-line text
  text(
    name: 'RELEASE_NOTES',
    defaultValue: '',
    description: 'What changed?'
  )

  // Dropdown list
  choice(
    name: 'ENVIRONMENT',
    choices: ['dev', 'staging', 'prod'],
    description: 'Target environment'
  )

  // True/False checkbox
  booleanParam(
    name: 'SKIP_TESTS',
    defaultValue: false,
    description: 'Skip test stage?'
  )

  // File upload (needs File Param plugin)
  file(
    name: 'CONFIG_FILE',
    description: 'Upload config file'
  )

  // Password (hidden in UI)
  password(
    name: 'SECRET',
    defaultValue: '',
    description: 'Secret value'
  )

  // Branch selector (needs Git Parameter plugin)
  gitParameter(
    name: 'BRANCH_NAME',
    type: 'PT_BRANCH',
    defaultValue: 'main',
    description: 'Select branch'
  )

}
Using Parameters in Pipeline
Accessing params.* 
stages {
  stage('Checkout') {
    steps {
      git branch: "${params.BRANCH_NAME}",
          url: 'https://github.com/org/repo.git'
    }
  }

  stage('Test') {
    when {
      // Skip this stage if param is true
      expression { return !params.SKIP_TESTS }
    }
    steps {
      sh "npx playwright test --project=${params.ENVIRONMENT}"
    }
  }

  stage('Deploy') {
    steps {
      script {
        // Use in Groovy logic
        if (params.ENVIRONMENT == 'prod') {
          input 'Confirm production deploy?'
        }
        sh "./deploy.sh ${params.ENVIRONMENT} ${params.VERSION}"
      }
    }
  }
}
⚠ Note
Parameters are only available after the first build. Jenkins needs one run to register them. Use defaultValue to handle the first run safely.
07

Triggers

cron — Schedule-based
Run at fixed times using cron syntax
triggers {
  cron('H 2 * * 1-5')   // 2AM Mon-Fri
  cron('H/15 * * * *')  // Every 15 min
  cron('0 8 * * 1')     // 8AM every Monday
}
Format: MIN HOUR DOM MON DOW  ·  H = hash (avoid thundering herd)
pollSCM — Polling-based
Jenkins checks Git for changes on a schedule
triggers {
  // Check every 5 minutes
  pollSCM('H/5 * * * *')
}
githubPush — Webhook-based
GitHub notifies Jenkins instantly on push
triggers {
  githubPush()
  // Requires GitHub plugin + webhook config
}
upstream — Job-based
Trigger when another job finishes
triggers {
  upstream(
    upstreamProjects: 'build-job,package-job',
    threshold: hudson.model.Result.SUCCESS
  )
}
Cron Quick Reference
ExpressionMeaning
H * * * *Once per hour
H H * * *Once per day
H H * * 0Once per week (Sunday)
H H 1 * *Once per month
H/30 * * * *Every 30 minutes
H 0-7 * * *Once in first 8 hours
H 9 * * 1-59AM on weekdays
@midnightOnce a day around midnight
@weeklyOnce a week
@monthlyOnce a month
💡 Use H instead of fixed numbers
H spreads load across Jenkins. Instead of 0 2 * * * (all jobs at exactly 2AM), use H 2 * * * (each job runs at a different minute within the 2AM hour).
08

Post Conditions

The post{} block runs AFTER stages complete. Can be at pipeline level or stage level.

always
Runs no matter what — success, failure, or abort. Use for cleanup/reporting.
success
Only when build result is SUCCESS. Good for deployment notifications.
failure
Only on FAILURE. Send alerts, create tickets, rollback deployments.
unstable
Result is UNSTABLE (tests failed but build passed). Common with test reports.
aborted
Manually stopped by a user. Good for cleanup of running processes.
changed
Result changed from the previous build (e.g. was success, now fail).
fixed
Was failing, now passing. Great for "it's back!" notifications.
regression
Was passing, now failing. Immediate alert trigger.
cleanup
Always runs LAST, after all other post conditions. Use for final cleanup.
Real-world post{} example 
post {
  always {
    // Always publish test results
    junit '**/test-results/*.xml'
    publishHTML([reportDir: 'playwright-report', reportFiles: 'index.html', reportName: 'E2E Report'])
  }
  success {
    slackSend(color: 'good', message: "✅ Build #${BUILD_NUMBER} passed on ${GIT_BRANCH}")
  }
  failure {
    slackSend(color: 'danger', message: "❌ Build #${BUILD_NUMBER} FAILED — ${BUILD_URL}")
    mail(to: 'qa-team@company.com', subject: "Build Failed: ${JOB_NAME}", body: "Check: ${BUILD_URL}")
  }
  fixed {
    slackSend(color: 'good', message: "🟢 Build is back to green!")
  }
  cleanup {
    sh 'docker system prune -f'
    cleanWs() // Clean workspace plugin
  }
}

When / Conditionals

The when{} block controls whether a stage runs. All conditions must be true (AND logic by default).

ConditionSyntaxUse Case
Branch namebranch 'main'Only on main branch
Branch patternbranch 'feature/*'All feature branches
Environment varenvironment name:'X', value:'Y'Env-based condition
Expressionexpression { return params.X == 'Y' }Any Groovy logic
Change typechangeRequest()Only on PRs
Tagtag 'v*'Only on version tags
Notnot { branch 'main' }Invert any condition
All of (AND)allOf { ... }Multiple must be true
Any of (OR)anyOf { ... }At least one must be true
File existsfileExists 'config.yaml'Only if file present
Build causetriggeredBy 'TimerTrigger'Only if triggered by cron
when{} Examples 
// Only deploy on main branch
stage('Deploy Prod') {
  when { branch 'main' }
  steps { sh './deploy.sh prod' }
}

// Run only if param is set
stage('Smoke Tests') {
  when {
    expression { params.RUN_SMOKE == true }
  }
  steps { sh 'npm run smoke' }
}

// Combined conditions (AND)
stage('Release') {
  when {
    allOf {
      branch 'main'
      environment name: 'CI', value: 'true'
    }
  }
}

// OR condition
stage('Notify') {
  when {
    anyOf {
      branch 'main'
      branch 'release/*'
    }
  }
}

// Only on Pull Requests
stage('PR Check') {
  when { changeRequest() }
  steps { sh 'npm run lint' }
}

Parallel Execution

Run multiple stages simultaneously to drastically reduce pipeline execution time.

Parallel Stages 
stage('Test Suite') {
  parallel {

    stage('Chrome Tests') {
      agent { label 'agent-1' }
      steps {
        sh 'npx playwright test --project=chromium'
      }
    }

    stage('Firefox Tests') {
      agent { label 'agent-2' }
      steps {
        sh 'npx playwright test --project=firefox'
      }
    }

    stage('Safari Tests') {
      agent { label 'agent-3' }
      steps {
        sh 'npx playwright test --project=webkit'
      }
    }

  }
}

// Fail fast: stop all if one fails
stage('Parallel with Failfast') {
  failFast true
  parallel {
    stage('A') { steps { sh 'test-a.sh' } }
    stage('B') { steps { sh 'test-b.sh' } }
  }
}
Dynamic Parallel (Matrix)
matrix{} — Auto-generate parallel stages 
stage('Cross-Browser Tests') {
  matrix {
    axes {
      axis {
        name: 'BROWSER'
        values: 'chromium', 'firefox', 'webkit'
      }
      axis {
        name: 'OS'
        values: 'linux', 'windows'
      }
    }

    // Exclude specific combos
    excludes {
      exclude {
        axis { name: 'BROWSER'; values: 'webkit' }
        axis { name: 'OS'; values: 'linux' }
      }
    }

    stages {
      stage('Run') {
        steps {
          sh "npx playwright test --project=${BROWSER}"
        }
      }
    }
  }
}
⚡ Speed Impact
Running 3 browser suites in parallel cuts test time by ~66%. A 30-minute suite becomes ~10 minutes.

Shared Libraries

Shared Libraries let you extract reusable pipeline code into a separate Git repo and import it across all your Jenkinsfiles — DRY principle for CI/CD.

Library Repo Structure
jenkins-shared-library/ 
jenkins-shared-library/
├── vars/
│   ├── deployApp.groovy      ← Global functions
│   ├── runPlaywright.groovy
│   └── sendSlack.groovy
├── src/
│   └── org/company/
│       └── Utils.groovy      ← Classes
└── resources/
    └── scripts/
        └── setup.sh          ← Non-Groovy files
vars/runPlaywright.groovy 
// Define a reusable function
def call(String browser = 'chromium', String env = 'staging') {
  sh """
    npm install
    npx playwright install
    npx playwright test \\
      --project=${browser} \\
      --reporter=html
  """
  publishHTML([
    reportDir: 'playwright-report',
    reportFiles: 'index.html'
  ])
}
Using in Jenkinsfile
Jenkinsfile · Using Shared Library 
// Import at the top (versioned!)
@Library('jenkins-shared-library@main') _

pipeline {
  agent any

  stages {
    stage('Test') {
      steps {
        // Call the shared function
        runPlaywright('chromium', 'staging')
      }
    }

    stage('Deploy') {
      steps {
        deployApp('production')
      }
    }
  }
}

// OR: Import multiple libraries
@Library(['shared-lib@v1.2', 'tools-lib']) _
📌 Setup in Jenkins
Go to Manage Jenkins → System → Global Pipeline Libraries and add your library repo URL, default version, and a name to reference it with @Library.

  // DOCKER_CREDS_USR and DOCKER_CREDS_PSW

withCredentials{} — In Steps
All Credential Binding Types 
steps {
  withCredentials([

    // Secret text
    string(credentialsId: 'my-token', variable: 'TOKEN'),

    // Username + Password
    usernamePassword(
      credentialsId: 'docker-creds',
      usernameVariable: 'DOCKER_USER',
      passwordVariable: 'DOCKER_PASS'
    ),

    // SSH Key
    sshUserPrivateKey(
      credentialsId: 'deploy-key',
      keyFileVariable: 'SSH_KEY'
    ),

    // Secret File
    file(credentialsId: 'env-file', variable: 'ENV_FILE'),

    // AWS
    aws(credentialsId: 'aws-creds',
        accessKeyVariable: 'AWS_ACCESS_KEY',
        secretKeyVariable: 'AWS_SECRET_KEY')

  ]) {
    sh """
      echo ${TOKEN} | docker login -u ${DOCKER_USER} --password-stdin
      ssh -i ${SSH_KEY} user@server 'deploy.sh'
    """
  }
}

QA / SDET Pipeline — Full Example

A production-ready Playwright pipeline with parameters, parallel execution, reporting, and notifications.

Jenkinsfile · Production QA Pipeline 
pipeline {
  agent {
    docker { image 'mcr.microsoft.com/playwright:v1.40.0-jammy' }
  }

  parameters {
    gitParameter(name: 'BRANCH_NAME', type: 'PT_BRANCH', defaultValue: 'main')
    choice(name: 'ENVIRONMENT', choices: ['staging', 'qa', 'prod'], description: 'Target environment')
    choice(name: 'SUITE', choices: ['regression', 'smoke', 'sanity'], description: 'Test suite')
    booleanParam(name: 'PARALLEL', defaultValue: true, description: 'Run browsers in parallel?')
  }

  environment {
    BASE_URL     = "https://${params.ENVIRONMENT}.myapp.com"
    REPORT_DIR   = 'playwright-report'
    SLACK_CHANNEL= '#qa-alerts'
  }

  triggers {
    // Nightly regression at 1AM Mon-Fri
    cron('H 1 * * 1-5')
  }

  options {
    timeout(time: 2, unit: 'HOURS')
    buildDiscarder(logRotator(numToKeepStr: '30'))
    timestamps()
  }

  stages {

    stage('Checkout') {
      steps {
        git branch: "${params.BRANCH_NAME}", url: 'https://github.com/org/app.git'
      }
    }

    stage('Install') {
      steps {
        sh 'npm ci'
      }
    }

    stage('Tests — Sequential') {
      when {
        expression { !params.PARALLEL }
      }
      steps {
        sh """
          BASE_URL=${BASE_URL} npx playwright test \\
            --grep=@${params.SUITE} \\
            --reporter=html,junit
        """
      }
    }

    stage('Tests — Parallel') {
      when {
        expression { params.PARALLEL == true }
      }
      parallel {
        stage('Chromium') {
          steps {
            sh "BASE_URL=${BASE_URL} npx playwright test --project=chromium --grep=@${params.SUITE}"
          }
        }
        stage('Firefox') {
          steps {
            sh "BASE_URL=${BASE_URL} npx playwright test --project=firefox --grep=@${params.SUITE}"
          }
        }
        stage('WebKit') {
          steps {
            sh "BASE_URL=${BASE_URL} npx playwright test --project=webkit --grep=@${params.SUITE}"
          }
        }
      }
    }

    stage('Deploy to Prod') {
      when {
        allOf {
          branch 'main'
          expression { params.ENVIRONMENT == 'prod' }
        }
      }
      steps {
        input message: 'Approve production deployment?', ok: 'Deploy'
        sh './deploy.sh prod'
      }
    }

  }

  post {
    always {
      junit '**/junit-results.xml'
      publishHTML([
        allowMissing: false,
        reportDir: "${REPORT_DIR}",
        reportFiles: 'index.html',
        reportName: 'Playwright Report',
        keepAll: true
      ])
    }
    success {
      slackSend(
        channel: "${SLACK_CHANNEL}",
        color: 'good',
        message: "✅ *${params.SUITE}* passed on *${params.ENVIRONMENT}* | Branch: ${params.BRANCH_NAME} | Build: #${BUILD_NUMBER} | ${BUILD_URL}"
      )
    }
    failure {
      slackSend(
        channel: "${SLACK_CHANNEL}",
        color: 'danger',
        message: "❌ *${params.SUITE}* FAILED on *${params.ENVIRONMENT}* | Build: #${BUILD_NUMBER} | ${BUILD_URL}"
      )
    }
    cleanup {
      cleanWs()
    }
  }
}

Scripted vs Declarative

✅ Declarative (Recommended)
pipeline {
  agent any

  stages {
    stage('Build') {
      steps {
        sh 'mvn package'
      }
    }

    stage('Test') {
      steps {
        sh 'mvn test'
      }
    }
  }

  post {
    failure {
      mail to: 'qa@co.com'
    }
  }
}
Structured Readable Validated post{} support
⚡ Scripted (Advanced)
node {

  stage('Build') {
    sh 'mvn package'
  }

  stage('Test') {
    try {
      sh 'mvn test'
    } catch (e) {
      // Full Groovy exception handling
      currentBuild.result = 'UNSTABLE'
      throw e
    } finally {
      // Always runs
      junit '**/*.xml'
    }
  }

}
Full Groovy Flexible loops Complex logic Harder to read
💡 Best Practice
Use Declarative by default. Drop into script{} blocks inside Declarative when you need Groovy logic. Never start a new project with pure Scripted syntax.
