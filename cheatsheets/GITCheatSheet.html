Git Fundamentals

Git is a distributed version control system — every clone is a full repository with complete history. No single point of failure.

Core Mental Model
Git tracks snapshots, not diffs. Each commit is a snapshot of your entire project at that point in time. Unchanged files are stored as references to the previous version.
The Three States:
Working Dir Your actual files on disk. This is where you edit code. Modified
Staging Area The index — a preview of your next commit. git add moves files here. Staged
Repository The .git directory. Committed snapshots live here permanently. Committed
Git Object Types:
Blob
File content. Stored as compressed binary. Identified by SHA-1 hash.
Tree
Directory listing. Maps filenames to blobs and other trees.
Commit
Snapshot pointer. Links to tree + parent commit(s) + metadata.
Tag
Named reference to a commit. Annotated tags include message + author.
💡 Everything is a hash
Every object in Git is identified by a SHA-1 hash (40 hex characters). Branches, tags, and HEAD are just pointers to these hashes.

Every Git interaction follows the same core loop: edit files, stage what you want, commit a snapshot, and push to share. Understanding this flow is the key to never feeling lost in Git.

📋 Stage git add → staging area
💾 Commit git commit → local repo
🚀 Push git push → remote repo
🔀 PR / MR Code review & merge

Setup & Configuration

Before your first commit, Git needs to know who you are. Configuration is layered — system-wide, per-user, and per-repo — with each level overriding the one above it.

Initial Setup
Terminal · First-time config 
# Identity (required for commits)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# Default branch name
git config --global init.defaultBranch main

# Default editor
git config --global core.editor "code --wait"

# Merge tool
git config --global merge.tool vscode

# Enable colored output
git config --global color.ui auto

# Store credentials
git config --global credential.helper store

# Line endings (cross-platform)
git config --global core.autocrlf input   # Mac/Linux
git config --global core.autocrlf true    # Windows
Config Scopes & Inspection
ScopeFlagLocation

Global (user)--global~/.gitconfig
Local (repo)--local.git/config
Worktree--worktree.git/config.worktree
Inspect & Init

git config --list

# View specific value
git config user.name

# Initialize a new repo
git init
git init my-project              # create dir + init

# Clone existing repo
git clone <url>

git clone --depth 1 <url>      # shallow clone
git clone --branch dev <url>   # specific branch
git clone --recurse-submodules <url>

Staging & Committing

The staging area is Git's superpower — it lets you craft exactly the snapshot you want before committing. Stage selectively with git add -p, then commit with a clear message. This two-step process is what makes Git commits meaningful.

git add — Stage Changes
CommandWhat It Does
git add <file>Stage specific file
git add .Stage all changes in current dir
git add -AStage ALL changes (entire repo)
git add -pInteractive: stage hunks selectively
git add *.jsStage by pattern (glob)
git add -uStage modified & deleted (not new)
💡 Use git add -p daily
The -p (patch) flag lets you stage individual hunks within a file. This is the secret to clean, atomic commits.
git status — Check State
git status               # full status
git status -s             # short format
git status -sb            # short + branch info

# Short status symbols:
# M  = modified     A = added      D = deleted
# R  = renamed      C = copied     U = unmerged
# ?? = untracked    !! = ignored
git commit — Save Snapshot
CommandWhat It Does
git commit -m "msg"Commit with inline message
git commitOpen editor for message
git commit -am "msg"Stage tracked + commit (skip add)
git commit --amendModify last commit
git commit --amend -m "new"Rewrite last commit message
git commit --amend --no-editAdd staged files to last commit
git commit --allow-emptyCreate empty commit (CI trigger)
git commit -SGPG-sign the commit
⚠ Commit message convention
type(scope): description

feat fix docs refactor perf test chore ci
Conventional Commits 
git commit -m "feat(auth): add OAuth2 login flow"
git commit -m "fix(api): handle null response in /users"
git commit -m "docs: update API migration guide"
git commit -m "refactor!: rename User to Account"  # breaking

Branching & Merging

Branches are Git's killer feature — they're lightweight pointers to commits, costing nearly zero to create. Use them for every feature, fix, and experiment. Merging brings the work back together when you're done.

Branch Commands
CommandWhat It Does
git branchList local branches
git branch -aList all (local + remote)
git branch -vList with last commit
git branch <name>Create branch (don't switch)
git branch -d <name>Delete (safe — merged only)
git branch -D <name>Force delete (even unmerged)
git branch -m <new>Rename current branch
git branch -m <old> <new>Rename any branch
git branch --mergedBranches merged into current
git branch --no-mergedBranches NOT merged
Switching Branches
# Modern (Git 2.23+)
git switch main                   # switch to existing
git switch -c feature/auth        # create + switch
git switch -c fix/bug origin/main  # from remote
git switch -                       # previous branch

# Classic
git checkout main
git checkout -b feature/auth
Merging
Merge Strategies 
# Standard merge (creates merge commit if diverged)
git merge feature/auth

# Fast-forward only (linear history)
git merge --ff-only feature/auth

# Always create merge commit
git merge --no-ff feature/auth

# Squash all commits into one
git merge --squash feature/auth
git commit -m "feat: add auth module"

# Abort a conflicted merge
git merge --abort

# Continue after resolving conflicts
git add .
git merge --continue
gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" commit id: "D" checkout main commit id: "E" merge feature id: "Merge" tag: "v1.0" commit id: "F"
Merge vs Rebase vs Squash
Merge: preserves full history. Rebase: linear history, rewrites commits. Squash: combines into single commit. Choose based on team convention.

Remote Operations

Remotes are references to repositories hosted elsewhere (GitHub, GitLab, Bitbucket). They're how you collaborate — fetch downloads changes without touching your work, pull integrates them, and push shares yours.

Managing Remotes
CommandWhat It Does
git remote -vList remotes with URLs
git remote add origin <url>Add a remote
git remote rename old newRename a remote
git remote remove <name>Remove a remote
git remote set-url origin <url>Change remote URL
git remote show originDetailed remote info
git remote prune originClean stale remote branches
Fetch vs Pull
# Fetch — download changes, don't merge
git fetch                          # fetch from origin
git fetch --all                    # all remotes
git fetch --prune                  # + clean deleted
git fetch origin main              # specific branch

# Pull = fetch + merge
git pull                           # current branch
git pull --rebase                  # fetch + rebase
git pull --rebase=interactive      # interactive
git pull origin main               # specific
Push Commands
git push — All Variants 
# Push current branch
git push

# First push — set upstream tracking
git push -u origin feature/auth

# Push all branches
git push --all

# Push tags
git push --tags

# Force push (CAREFUL!)
git push --force               # overwrites remote
git push --force-with-lease     # safe force push ✅

# Delete remote branch
git push origin --delete feature/old
git push origin :feature/old    # shorthand
🚨 Force Push Safety
Never force-push to shared branches (main, develop). Always use --force-with-lease — it fails if someone else pushed since your last fetch.

Undoing Changes — The Survival Guide

The most important section in this cheat sheet. Git has multiple "undo" mechanisms — each for a different situation.

📝 Discard Working Dir Changes
# Modern (Git 2.23+)
git restore file.js          # discard changes
git restore .                # discard ALL

# Classic
git checkout -- file.js
📋 Unstage Files
# Modern
git restore --staged file.js
git restore --staged .

# Classic
git reset HEAD file.js
💾 Amend Last Commit
# Fix message
git commit --amend -m "new msg"

# Add forgotten files
git add forgotten.js
git commit --amend --no-edit
⏪ Reset (Rewrite History)
# Soft: uncommit, keep staged
git reset --soft HEAD~1

# Mixed (default): uncommit + unstage
git reset HEAD~1

# Hard: uncommit + unstage + DELETE
git reset --hard HEAD~1
↩️ Revert (Safe Undo)
# Create inverse commit (safe for shared)
git revert abc123
git revert HEAD               # undo last
git revert HEAD~3..HEAD       # undo range
git revert --no-commit abc123 # stage only
🗑️ Remove from Git
# Remove file from Git AND disk
git rm file.js

# Remove from Git, KEEP on disk
git rm --cached file.js

# Remove directory
git rm -r --cached dist/

Tagging & Releases

Tags mark specific commits as important — usually for releases. Unlike branches, tags don't move. Use lightweight tags for temporary markers and annotated tags for official releases with messages, authors, and optional GPG signatures.

Tag Types & Creation
Tags — Lightweight vs Annotated 
# Lightweight tag (just a pointer)
git tag v1.0.0

# Annotated tag (recommended — has metadata)
git tag -a v1.0.0 -m "Release 1.0.0"

# Tag a specific commit
git tag -a v0.9.0 abc123 -m "Retroactive tag"

# GPG-signed tag
git tag -s v1.0.0 -m "Signed release"

# List tags
git tag                         # all tags
git tag -l "v1.*"               # with pattern
git tag -l --sort=-version:refname # sorted
Push, Delete & Checkout Tags
Remote Tag Operations 
# Push specific tag
git push origin v1.0.0

# Push ALL tags
git push --tags

# Push only annotated tags
git push --follow-tags

# Delete local tag
git tag -d v1.0.0

# Delete remote tag
git push origin --delete v1.0.0

# Checkout tag (detached HEAD)
git checkout v1.0.0

# Create branch from tag
git switch -c hotfix/v1 v1.0.0
Semantic Versioning
MAJOR.MINOR.PATCH — v1.2.3
MAJOR = breaking changes, MINOR = new features, PATCH = bug fixes

Rebase & Cherry-Pick

Rebase replays your commits on top of another branch — creating a clean, linear history without merge commits. Cherry-pick copies individual commits between branches. Both are powerful history-editing tools, but rebase should never be used on shared branches.

git rebase — Rewrite History
Rebase Commands 
# Rebase current branch onto main
git rebase main

# Interactive rebase (last 4 commits)
git rebase -i HEAD~4

# Interactive rebase keywords:
#   pick   = keep commit as-is
#   reword = change message
#   edit   = pause to amend
#   squash = merge into previous
#   fixup  = squash + discard message
#   drop   = remove commit entirely

# Continue after resolving conflict
git rebase --continue

# Abort and return to pre-rebase state
git rebase --abort

# Skip conflicted commit
git rebase --skip

# Rebase onto different base
git rebase --onto main feature-old feature-new
gitGraph commit id: "A" commit id: "B" branch feature checkout feature commit id: "C" commit id: "D" checkout main commit id: "E" commit id: "F" checkout feature commit id: "C'" type: HIGHLIGHT commit id: "D'" type: HIGHLIGHT
git cherry-pick — Copy Specific Commits
Cherry-Pick Commands 
# Apply a specific commit to current branch
git cherry-pick abc123

# Multiple commits
git cherry-pick abc123 def456

# Range of commits
git cherry-pick abc123..def456

# Don't commit (just stage changes)
git cherry-pick --no-commit abc123

# Abort on conflict
git cherry-pick --abort

# Continue after resolving
git cherry-pick --continue
⚠ The Golden Rule of Rebase
Never rebase commits that have been pushed to a shared remote. Rebase rewrites commit hashes — your teammates will have orphaned commits if you rebase shared history.
💡 When to Use Each
Rebase: keep feature branch up-to-date with main (before PR).
Cherry-pick: apply a specific hotfix from one branch to another.
Merge: integrate completed feature into main.

Advanced Commands

These are the commands that separate Git users from Git power users. From binary-searching for bugs with bisect to recovering "lost" commits with reflog — these tools save hours when things get complicated.

🔍
git bisect
Binary search for bugs
git bisect start
git bisect bad                  # current is bad
git bisect good v1.0            # this was good
# Git checks out midpoint — test it
git bisect good                 # or "bad"
# ... repeat until found ...
git bisect reset                # done

# Automated bisect with a test script
git bisect run npm test
🕐
git reflog
Recovery safety net
# Show all HEAD movements (even "lost" commits)
git reflog

# Reflog for specific branch
git reflog show feature

# Recover "lost" commit after reset
git reset --hard HEAD@{3}

# Recover deleted branch
git branch recovered HEAD@{5}
Reflog is your undo button for undos. Entries expire after 90 days.
🌳
git worktree
Multiple working directories
# Create worktree for another branch
git worktree add ../hotfix hotfix/auth

# Create worktree + new branch
git worktree add -b feature/new ../feat

# List worktrees
git worktree list

# Remove worktree
git worktree remove ../hotfix

📦

Nested repositories

Branching Strategies — Visual Guide

How your team uses branches defines your workflow. There's no single "correct" strategy — pick one that matches your release cadence, team size, and deployment model. Here are the three most popular approaches, visualized with Mermaid git graphs.

🌿
GitHub Flow
Simple & effective
Branches: main + short-lived feature
Rule: main is always deployable
Flow: branch → commit → PR → review → merge → deploy

Best for: SaaS, web apps, small teams
🔀
GitFlow
Structured releases
Branches: main, develop, feature/*, release/*, hotfix/*
Rule: Never commit directly to main or develop
Flow: feature → develop → release → main

Best for: Versioned software, mobile apps
🚀
Trunk-Based
Continuous delivery
Branches: main (trunk) + very short-lived branches
Rule: Merge to main daily. Use feature flags.
Flow: branch → small PR → merge same day

Best for: Large teams, CI/CD, Google/Meta-style

10 Common Git Problems & Solutions

Real-world disasters you will face — and exactly how to fix them. Multiple solutions ranked from safest to most aggressive.

01
Accidentally committed to the wrong branch
You made commits on main instead of a feature branch
Solution A — Move commits to new branch (safest)
# Create new branch pointing to current HEAD (keeps your commits)
git branch feature/my-work

# Reset main back to before your commits
git reset --hard origin/main

# Switch to your new branch — your work is there!
git switch feature/my-work
Solution B — If you haven't committed yet
git stash
git switch -c feature/my-work
git stash pop
02
Need to undo the last commit but keep changes
You committed too early or with wrong message
Solution A — Soft reset (keep files staged)
git reset --soft HEAD~1      # undo commit, keep staged
Solution B — Mixed reset (keep files, unstage)
git reset HEAD~1              # undo commit + unstage, keep files
Solution C — Just fix the message
git commit --amend -m "corrected message"
03
Merge conflicts everywhere
Two branches modified the same lines
Solution A — Resolve manually
# 1. See conflicted files
git status

# 2. Open and edit files — look for conflict markers:
# <<<<<<< HEAD
#   your changes
# =======
#   their changes
# >>>>>>> feature-branch

# 3. Remove markers, keep the code you want

# 4. Mark resolved and continue
git add resolved-file.js
git merge --continue         # or git commit
Solution B — Accept one side entirely
git checkout --theirs file.js  # keep incoming branch's version
git checkout --ours file.js    # keep current branch's version
git add file.js
Solution C — Abort and try rebase instead
git merge --abort
git rebase main               # resolve conflicts one commit at a time
04
Detached HEAD state
You checked out a commit or tag and now you're lost
Solution A — Go back to a branch
git switch main              # return to branch
Solution B — Save your work from detached HEAD
# If you made commits in detached HEAD — save them!
git switch -c rescue-branch  # create branch at current position
05
Accidentally deleted a branch
You ran git branch -D and regret it
Solution A — Recover via reflog
# Find the commit hash the branch pointed to
git reflog | grep "branch-name"

# Recreate the branch at that commit
git branch branch-name abc123
Solution B — If it existed on remote
git fetch origin
git switch branch-name       # auto-tracks remote
06
Need to remove a file from Git but keep it locally
You committed .env, logs, or build artifacts
Solution — Untrack without deleting
# Remove from Git, keep on disk
git rm --cached .env
git rm -r --cached node_modules/

# Add to .gitignore
echo ".env" >> .gitignore

# Commit the removal
git add .gitignore
git commit -m "chore: untrack .env, add to gitignore"
07
Committed sensitive data (secrets, passwords, API keys)
Critical security issue — secrets are now in Git history
🚨 Critical
Even if you delete the file in a new commit, the secret is still in history. You MUST rewrite history or consider the credential compromised.
Solution A — git filter-repo (recommended)
# Install git-filter-repo (pip install git-filter-repo)
git filter-repo --path secrets.env --invert-paths
git push --force --all
Solution B — BFG Repo-Cleaner (faster for large repos)
bfg --delete-files secrets.env
git reflog expire --expire=now --all
git gc --prune=now --aggressive
git push --force --all
Solution C — If just the last commit
git reset --soft HEAD~1
# Remove the secret file, add to .gitignore
git rm --cached secrets.env
git commit -m "remove secrets"
git push --force-with-lease    # if already pushed
Always rotate the compromised credential immediately regardless of method used.
08
Need to squash messy commits before merging PR
Your feature branch has 15 "WIP" and "fix typo" commits
Solution A — Interactive rebase
# Squash last 5 commits
git rebase -i HEAD~5

# In editor — change "pick" to "squash" (or "s") for all except first:
# pick   abc1234 feat: add login form
# squash def5678 wip
# squash ghi9012 fix typo
# squash jkl3456 more fixes

Need to find which commit introduced a bug
Something broke and you don't know when
Solution A — git bisect (binary search)
git bisect start
git bisect bad                  # current commit has the bug
git bisect good v1.2.0          # this version was fine

# Git checks out the midpoint. Test it, then:
git bisect good                 # if this commit is fine
git bisect bad                  # if this commit has the bug

# Repeat until Git narrows down to THE commit
git bisect reset                # return to original branch
Solution B — Automated bisect
git bisect start HEAD v1.2.0
git bisect run npm test        # fully automatic!

Aliases & Productivity

Git aliases turn multi-word commands into short, memorable shortcuts. Set them once and save thousands of keystrokes over your career. These are battle-tested aliases used by senior engineers daily.

Must-Have Git Aliases
~/.gitconfig · [alias] 
# Quick shortcuts
git config --global alias.s "status -sb"
git config --global alias.co "checkout"
git config --global alias.br "branch"
git config --global alias.ci "commit"
git config --global alias.sw "switch"

# Log visualizations
git config --global alias.lg \
  "log --graph --oneline --decorate --all"
git config --global alias.ll \
  "log --pretty=format:'%C(yellow)%h %C(blue)%ad %C(green)%an%C(reset) %s' --date=short"

# Undo shortcuts
git config --global alias.undo "reset --soft HEAD~1"
git config --global alias.unstage "restore --staged ."
git config --global alias.discard "restore ."

# Workflow
git config --global alias.amend "commit --amend --no-edit"
git config --global alias.wip "commit -am 'WIP'"
git config --global alias.last "log -1 HEAD --stat"

# Cleanup merged branches
git config --global alias.cleanup \
  "!git branch --merged main | grep -v main | xargs git branch -d"
Power User Techniques
git switch - (dash)
Toggle to previous branch. Like cd - for Git.
git add -p (patch mode)
Stage individual hunks within a file for surgical commits.
git commit --fixup=<sha>
Auto-create a fixup commit; git rebase -i --autosquash to merge it.
git diff --word-diff
See inline word-level changes instead of full-line diffs.
git log -S "text" (pickaxe)
Find commits that added or removed specific string from codebase.
git stash push -m "msg" -p
Named, partial stash — only stash select hunks with a description.
💡 Speed Tip
Set git config --global pull.rebase true to auto-rebase on every pull. Keeps history clean without thinking about it.
